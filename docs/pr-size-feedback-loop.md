# PR Size Feedback Loop

LLM による PR サイズ分類を、チームの暗黙知で補正し、
分類精度の向上とアクション自動化を段階的に実現する計画。

## 理論的背景: 暗黙知と知識創造

### 暗黙知とは

マイケル・ポランニーは「**私たちは語れるよりも多くのことを知っている**」(The Tacit Dimension, 1966) と述べ、
言語化が困難だが行動に埋め込まれた知識を **暗黙知 (tacit knowledge)** と定義した。

コードレビューにおける暗黙知の例:

- 「認証周りは 1 行でも怖い」— リスク領域の重み付け
- 「このリポはテスト網羅されてるから安心」— コンテキスト依存の信頼度
- 「lock file の差分は見なくていい」— 機械的変更の判断

これらは言語化されないまま各レビュアーの頭の中に留まっている。
LLM classifier はこの暗黙知にアクセスできないため、人間の感覚とズレが生じる。

### SECI モデルによるフレームワーク

野中郁次郎の **SECI モデル** (1995) は、暗黙知と形式知がどのように変換され組織知が創造されるかを示す。
このフィードバックループは SECI の 4 プロセスに対応する:

| SECI プロセス                | 本計画での対応                                                    | Phase |
| ---------------------------- | ----------------------------------------------------------------- | ----- |
| **共同化** (Socialization)   | チーム内でレビューを通じて「この領域は怖い」という感覚を共有      | 前提  |
| **表出化** (Externalization) | フィードバック UI + AI ドラフトで暗黙知を reason テキストに言語化 | 1     |
| **連結化** (Combination)     | フィードバックを集約しルール化、classify プロンプトに注入         | 2-3   |
| **内面化** (Internalization) | 改善された分類がチームの共通認識として定着                        | 4     |

Phase 1 の AI ドラフト機能は **表出化のコスト低減** が狙い。
「reason を書かない人が多い」問題は表出化の認知コストが高いことそのものであり、
LLM がドラフトを生成することでこのハードルを下げる。

参考:

- Polanyi, M. (1966). _The Tacit Dimension_
- Nonaka, I. & Takeuchi, H. (1995). _The Knowledge-Creating Company_
- [AI PRレビューでボトルネックを解消する](https://zenn.dev/pepabo/articles/ai-pr-review-bottleneck)

## PR サイズ定義

分類の軸は **レビューの認知負荷 (cognitive load)** と **影響範囲 (impact scope)**。
diff の行数ではなく「レビュアーがどれだけ頭を使うか」で判定する。

### 分類レベル

| Size   | 認知負荷   | 影響範囲                         | 判断基準                                           | 典型例                                                                           |
| ------ | ---------- | -------------------------------- | -------------------------------------------------- | -------------------------------------------------------------------------------- |
| **XS** | ほぼゼロ   | 局所的・機械的                   | 変更の意図を読む必要がない                         | typo修正, config値変更, lock file更新, revert, リリースPR, ファイル移動/リネーム |
| **S**  | 低い       | 単一の関心事                     | 1つのことだけ確認すれば良い                        | 小さなバグ修正, テスト追加, ドキュメント更新, feature flag切替                   |
| **M**  | 中程度     | 1コンポーネント内                | 1つのモジュールの文脈理解が必要。変更の境界が明確  | スコープの明確な新機能（1 endpoint, 1 component）, モジュール内リファクタ        |
| **L**  | 高い       | 複数コンポーネント or リスク領域 | 複数モジュール間の関係理解、または重要領域への変更 | DB schema + API + UI の横断変更, 認証/決済ロジック, 新サブシステム               |
| **XL** | 非常に高い | システム全体                     | アーキテクチャレベルの理解が必要                   | アーキテクチャ刷新, フレームワーク移行, 大規模リライト                           |

### 判断フロー

```text
1. 機械的か？ → Yes → XS or S（意図の読解不要なら XS、最小限必要なら S）
2. リスク領域か？（DB schema, 認証, 決済, セキュリティ） → Yes → 最低 L
3. 何コンポーネントに跨るか？
   - 1つの関心事 → S
   - 1コンポーネント内 → M
   - 複数コンポーネント → L
   - システム全体 → XL
4. diff量はタイブレーカーとして使う（隣接レベルで迷った時のみ）
```

### 設計方針

- **行数は判断基準にしない**: lock file 更新は数千行でも XS、認証ロジック変更は 10 行でも L
- **機械的変更を最優先で判定**: revert, リリース PR, codegen は diff 量に関係なく XS
- **リスク領域は格上げ**: DB schema, 認証, 決済, セキュリティに触れる変更は最低 L
- **diff量はタイブレーカー**: 認知負荷が隣接レベルで曖昧な場合のみ参考にする

## ゴール

1. **表出化の仕組みを作る** — フィードバックが自然に蓄積される UI
2. **連結化で精度向上** — 蓄積されたフィードバックで classify の精度を改善する
3. **内面化とアクション** — サイズ × リスクに基づくアクション推奨（自動マージ、分割提案、エース投入）

## Phase 1: 表出化 — フィードバック UI

**目的**: LLM 分類と人間の判断の差分を収集し、暗黙知を言語化するループを作る。

### データモデル

テナント DB の `pull_request_feedbacks` テーブル:

- `pullRequests.complexity` は上書きしない — 「LLM は XS と言ったが人間は M にした」という差分が価値
- 1 PR につき 1 フィードバック（上書き更新）
- 修正履歴で後の分析やルール抽出に使える

### UI

Dashboard / Ongoing テーブルの SizeBadge をクリック → Popover:

- 現在のサイズ表示（LLM 判定 + 理由 + リスク領域）
- 5 つのサイズボタン（XS / S / M / L / XL）から選択
- 理由テキストフィールド + 「AI Draft」ボタン
- 保存ボタン
- 修正済みバッジはアイコンで視覚的に区別

### AI ドラフト機能（表出化の支援）

理由の手入力はハードルが高く書かない人が多い（= 表出化コストが高い）。
「AI Draft」ボタンを押すと、LLM が PR 情報から修正理由を推測して下書きを入れる。

LLM に渡すコンテキスト:

- PR タイトル、ブランチ名、変更ファイル一覧、additions/deletions
- LLM の元の分類結果と理由（`complexity`, `complexityReason`, `riskAreas`）
- ユーザーが選択した修正後のサイズ
- レビューコメント、コミットメッセージ（`githubRawData` から取得）

プロンプトにはサイズ定義を注入し、定義の基準に基づいた理由を生成させる。
この推測自体が暗黙知の言語化（表出化）になり、Phase 2 のルール抽出の質を上げる。

### 実装場所

- スキーマ: `db/tenant.sql` — `pull_request_feedbacks` テーブル
- フィードバック保存: `app/routes/$orgSlug/pr-size-feedback.ts`
- AI ドラフト: `app/routes/$orgSlug/draft-feedback-reason.ts`
- UI: `app/routes/$orgSlug/+components/size-badge-popover.tsx`

## Phase 2: 連結化 — フィードバック集約とルール抽出

**目的**: 蓄積されたフィードバック（表出化された暗黙知）からリポジトリ/チーム単位のルールを自動生成する。

- リポジトリ単位でフィードバックを集約
- LLM で傾向を要約してルール化（例: 「このリポでは `src/auth/` 配下の変更は M 以上」）
- ルールはテナント DB に保存

## Phase 3: 連結化 → 内面化 — コンテキスト注入

**目的**: 抽出されたルールを classify プロンプトに注入し、精度を向上させる。

- `batch/lib/llm-classify.ts` のプロンプトにリポ固有ルールを追加
- フィードバック付き PR を few-shot example として使う
- 精度の変化をトラッキングする仕組み（フィードバック率の推移で計測）

## Phase 4: 内面化 — アクション推奨

**目的**: サイズとリスクに基づいて具体的なアクションを提案し、改善された分類をチームの行動に定着させる。

### アクションマトリクス

```text
              低リスク              高リスク
XS/S    → AI レビュー+自動マージ  → 人間レビュー（軽め）
M       → 通常レビュー            → シニアレビュー
L/XL    → 分割提案               → エース投入+同期レビュー
```

### 機能

- Dashboard に推奨アクションを表示
- XS/低リスク PR の自動マージ候補リスト
- L/XL PR への分割提案（LLM で分割案を生成）
- 高リスク PR へのエースアサイン提案

## 現行プロンプトの課題

現在 2 つの LLM プロンプトがあり、それぞれに課題がある。

### 分類プロンプト (`batch/lib/llm-classify.ts`)

PR を XS〜XL に分類する。Decision procedure（5 ステップ）とレベル定義を含む。

**課題**:

1. **行数目安が判断基準に混入している**
   - M: "Roughly 100-500 meaningful lines", L: "500-1500", XL: "1500+"
   - 上記「PR サイズ定義」の設計方針（行数は判断基準にしない）と矛盾
   - lock file 更新 3000 行を M〜L に誤分類する原因になりうる
2. **リスク領域の格上げが判断フローに組み込まれていない**
   - L の examples に "auth/payment/security" はあるが、Decision procedure に「リスク領域なら最低 L」のステップがない
   - 認証ロジック 10 行の変更が S に分類されるリスク
3. **影響範囲の軸が弱い**
   - Decision procedure の Step 3 は「how many distinct concerns」だが、コンポーネント間の依存や波及範囲への言及がない
4. **`risk_areas` が未定義で自由記述になっている**
   - スキーマ定義が `'Risk areas (e.g. "auth", "DB migration", "payment")'` のみ
   - 実際の出力: `deployment`(249件), `release management`(107件), `CI/CD pipeline`(69件) など本来リスクでないものが大量に付与
   - `DB migration`(45件) や `external API integration`(18件) など本当に重要なものがノイズに埋もれる
   - 改善案: リスク領域を閉じたリストに限定（`auth`, `DB schema/migration`, `payment/billing`, `security`, `external API`）し、該当なしは空配列を許可

### フィードバック理由ドラフトプロンプト (`app/routes/$orgSlug/draft-feedback-reason.ts`)

人間の修正理由を自動生成する。

**対応済み**: サイズ定義の注入、Gemini 3 prompting guide 準拠の構造化を実施。

### 改善方針（残課題）

- 分類プロンプトから行数目安を削除し、判断フローにリスク領域ステップを追加
- `risk_areas` を閉じたリスト（enum）に変更し、Decision procedure に「リスク領域なら最低 L」ステップを追加
- docs の「PR サイズ定義」を Single Source of Truth とし、分類プロンプトにも反映
- 将来的には Phase 2 で蓄積されたフィードバックも定義の改善に活用する

## 依存

- Phase 1 は現状の仕組みだけで着手可能
- Phase 2-3 はフィードバックが十分に蓄積されてから
- Phase 4 は GitHub API 連携（自動マージ、レビュアーアサイン）が必要

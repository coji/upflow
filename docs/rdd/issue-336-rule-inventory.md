## arch-review persona instruction 改訂提案

機械チェック（`機械化対象の優先順位` の推奨実装順 1〜5＝ Easy 群：エラーメッセージ、`~/`、`orgSlug` キー、DB/batch の日時 ISO）がリポジトリに定着し、grep / CI が green になり続ける前提で、`arch-review` ペルソナから毎回読ませる責務を外せる候補。

1. **`getErrorMessage` / `String(e)` 禁止** — grep が green ならレビュー確認から除外。
2. **`orgSlug` を含む PRPopover fetcher key** — 静的検査で担保可能。
3. **`~/` import エイリアス** — dependency-cruiser で自動化。
4. **DB への日時 ISO 8601（Z）保存／batch での GitHub ISO 無変換** — grep や migration レビューを CI に載せればレビュー範囲を縮小。
5. **DB 由来日時の `dayjs.utc`・SSR の `.tz`** — 完全自動は難しいが、(4) と合わせた grep で確認項目を限定できる。
6. **`onConflict` にフォーム由来 ID を載せない** — ルール化された静的解析が入ればペルソナ・チェックリストから削除。

逆に、(b) のコミュニケーション規範・設計トレードオフの説明・「やらない理由を言い切る」類は自動化が効きにくいため、ペルソナまたは人間レビューに残す。

# PR サイズ分類ガイド — 開発マネージャー向け

## これは何？

Upflow では、GitHub の Pull Request を **レビューの認知負荷** に基づいて XS〜XL の 5 段階に自動分類しています。この分類を使って「どの PR に注意を払うべきか」「レビュー負荷が偏っていないか」をダッシュボードで可視化しています。

現在この分類は LLM（Gemini）が自動で行っていますが、**人間の感覚とズレている** ケースがあります。

> 例: lock file だけの更新で 3000 行の diff → LLM は「L」と判定するが、実際のレビュー負荷はほぼゼロ（XS）

このズレを直すには、**実際にレビューしている皆さんの判断** が必要です。

## なぜ皆さんの協力が必要なのか

### レビューの「暗黙知」

レビュアーは PR を見た瞬間に「これは軽い」「これは怖い」と判断しています。例えば：

- 「認証周りは 1 行でも慎重にレビューする」
- 「このリポはテストが充実してるから安心」
- 「lock file の差分は飛ばして OK」

こうした判断はコードベースやチームの文脈に依存していて、LLM は知りません。**皆さんの頭の中にしかない知識**（暗黙知）を LLM に教えることで、分類の精度が上がります。

### フィードバックが分類を改善する仕組み

```
┌─────────────┐    ┌──────────────┐    ┌─────────────────┐
│ LLM が分類  │ →  │ 人間が補正   │ →  │ 補正データ蓄積  │
│ (自動)      │    │ (ダッシュボード) │    │ (フィードバック) │
└─────────────┘    └──────────────┘    └────────┬────────┘
                                                │
                   ┌──────────────┐              │
                   │ 分類精度向上 │ ←────────────┘
                   │ (プロンプト改善) │
                   └──────────────┘
```

皆さんが「この PR は LLM の判定と違って本当は M だよね」とフィードバックすると、そのデータが蓄積されます。蓄積されたフィードバックから「このリポではこういう変更は M 以上」といったルールを抽出し、LLM の分類プロンプトに反映していきます。

## 分類の基準

### 大原則

**diff の行数ではなく、レビュアーがどれだけ頭を使うか** で判定します。

- lock file 更新 3000 行 → レビュー不要 → **XS**
- 認証ロジック変更 10 行 → 慎重なレビューが必要 → **L**

### 5 段階の定義

| Size   | 認知負荷   | どんな変更？                               | 典型例                                                        |
| ------ | ---------- | ------------------------------------------ | ------------------------------------------------------------- |
| **XS** | ほぼゼロ   | 機械的。意図を読む必要がない               | typo 修正, config 値変更, lock file 更新, revert, リリース PR |
| **S**  | 低い       | 1 つのことだけ確認すればよい               | 小さなバグ修正, テスト追加, ドキュメント更新                  |
| **M**  | 中程度     | 1 つのモジュールの文脈理解が必要           | スコープの明確な新機能, モジュール内リファクタ                |
| **L**  | 高い       | 複数モジュールの関係理解、またはリスク領域 | DB + API + UI の横断変更, 認証/決済ロジック                   |
| **XL** | 非常に高い | アーキテクチャレベルの理解が必要           | アーキテクチャ刷新, フレームワーク移行                        |

### 判断の手順

迷ったときはこのフローで考えてください：

```
1. 機械的な変更か？
   → Yes → XS（意図の読解不要）or S（最小限の確認が必要）

2. リスク領域に触れているか？（DB schema, 認証, 決済, セキュリティ, 外部 API）
   → Yes → 最低 L

3. 何コンポーネントに跨るか？
   - 1 つの関心事 → S
   - 1 コンポーネント内 → M
   - 複数コンポーネント → L
   - システム全体 → XL

4. diff 量は隣接レベルで迷ったときだけ参考にする
```

### よくある判断例

| ケース                                  | LLM がよく間違える判定 | 正しい判定 | 理由                               |
| --------------------------------------- | ---------------------- | ---------- | ---------------------------------- |
| lock file のみの更新（数千行）          | M〜L                   | **XS**     | 機械的、レビュー不要               |
| リリース PR（複数 PR をまとめてマージ） | M〜L                   | **XS**     | 中身はレビュー済み                 |
| 認証ロジックの小さな変更（10 行）       | S                      | **L**      | リスク領域、慎重なレビューが必要   |
| DB マイグレーション + API 変更          | M                      | **L**      | 複数コンポーネント横断 + DB リスク |
| テストファイルの追加のみ                | M                      | **S**      | 単一の関心事、確認が容易           |

## 分類作業の進め方

### ダッシュボードでのフィードバック

1. Upflow のダッシュボード（Dashboard または Ongoing ページ）を開く
2. PR 一覧の **サイズバッジ**（XS/S/M/L/XL）をクリック
3. ポップオーバーが開く：
   - 上部に LLM の判定結果と理由が表示される
   - 5 つのサイズボタンから正しいと思うサイズを選択
   - 理由を入力（**AI Draft** ボタンを押すと LLM が下書きを生成してくれます）
   - **Save** で保存

### フィードバックのコツ

- **全部の PR にフィードバックする必要はありません**。LLM の判定が「違うな」と感じた PR だけで OK
- **理由は短くて大丈夫です**。「lock file だけ」「認証ロジック変更」程度で十分。迷ったら AI Draft を使ってください
- **正解は 1 つではありません**。隣接レベル（S と M の間など）で迷うのは自然です。自分がレビュアーとして感じる認知負荷で判断してください

### 特に見てほしい PR

以下のような PR は LLM が間違えやすいので、積極的にフィードバックをお願いします：

- **大量の diff だけどレビュー負荷が低い PR** — lock file 更新、自動生成コード、リネーム
- **diff は少ないけど重要な PR** — 認証、決済、DB スキーマ変更
- **リリース PR、revert PR** — 機械的だが diff が大きくなりがち
- **リファクタリング PR** — スコープによって M〜L に分かれる

## この先の展望

皆さんのフィードバックが蓄積されると：

1. **リポジトリ固有のルールを自動生成** — 「このリポの `src/auth/` 配下は M 以上」のようなルールが蓄積データから抽出される
2. **LLM の分類精度が向上** — 抽出されたルールが分類プロンプトに反映される
3. **アクション推奨** — サイズ × リスクに基づいて「この PR は自動マージ候補」「この PR はシニアレビューが必要」といった提案ができるようになる

最初のステップは **皆さんの「この判定おかしいな」というフィードバック** です。気軽にバッジをクリックして修正してください。

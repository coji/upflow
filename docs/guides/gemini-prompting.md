# Gemini 3 Prompting Best Practices

## プロンプト構造（順序が重要）

1. ゴール / 何をしてほしいか
2. 利用可能な入力 / 制約
3. 出力形式
4. 評価基準

モデルに「何を達成すればよいか」を最初に固定し、その後に制約と形式を与える。

## Temperature

- 厳密な抽出・分類: 低め
- 発想を広げたい草案: やや高め

用途に応じて変える。まずは deterministic 寄りを基本にする。

## 指示の書き方

### 広範囲な否定を避ける

- 悪い: 「曖昧にしないで」
- 良い: 「不明なら `unknown` を返す」

禁止だけを書くより、期待する挙動を具体化する。

### Grounding（提供情報への固定）

- 入力テキスト、表、ログ、仕様を明示する
- 推測で補わない条件を必要なら書く

モデルが外部知識や思い込みに流れないようにする。

### 2段階検証

重要な処理では、次の流れに分ける。

1. 下書きや抽出
2. 条件チェックや自己検証

特に分類や変換で有効。

## ペルソナ

- 抽出や変換では余計なペルソナは不要
- レビューや提案では、役割を短く明示するとぶれにくい

例:

- 「厳密なレビュアーとして」
- 「運用担当者向けに」

## 出力の詳細度

- 形式が重要なときは、詳細度より先に schema を固定する
- 箇条書き、JSON、表などを明示する

出力を短くしたい場合も、要件不足で曖昧にしない。

## Thinking Mode

- 複雑な判断では、内部で段階的に考えさせる前提で指示を構成する
- ただし、出力には不要な思考過程を要求しない

必要なのは「正しい結果」であって、長い独白ではない。

## 複数ソースの統合

- ソースごとに役割を分ける
- 矛盾時の優先順位を決める

例:

- 「仕様書を優先し、ログは補助情報とする」

## プロンプトの構造化: プレフィックス vs XML タグ

### シンプルなプロンプト → プレフィックス

短い指示、単一タスク、単一入力ならプレフィックスで十分。

### 複雑なプロンプト → XML タグ / デリミタ

入力と指示が長い場合、以下のように区切る。

```xml
<task>...</task>
<input>...</input>
<constraints>...</constraints>
<output_format>...</output_format>
```

混線を防ぎやすい。

### 使い分けの基準

- 短い: プレフィックス
- 長い / 複数入力 / 厳密な形式: 構造化タグ

## チェックリスト

- ゴールが最初に書かれているか
- 入力と制約が分離されているか
- 出力形式が明示されているか
- 推測してよい範囲が明確か
- 重要なら検証ステップがあるか

# マージキュー

## 要点

PR がマージ前に並ぶ順序付き列。最新の main に対して自動 rebase し、CI を走らせ、通った PR から順にマージする仕組み。「rebase してまた CI を待つ」という人手の繰り返しを消せる。アクティブな main を持つチームに効く。

## なぜ必要か

### マージレースの問題

PR A と PR B が同時期に承認されたとき:

```
状況: 両方とも origin/main の最新コミット X 時点で CI が green

A が先にマージ → main の最新は Y (= X + A の変更)
B はまだ X ベース
B を直接マージすると: B の差分が X の状態を前提にしている
                    → 実際のマージ結果は Y + B-on-X (期待と違う可能性)
```

正しくやるには B を Y にリベース → CI 再実行 → green 確認 → マージ。これを毎回手動でやると:

- レビュアーが承認したのに、リベース → CI 待ちで時間がさらに経つ
- その間に C が割り込んでまた最新でなくなる
- 著者が一日中 rebase ループに付き合うことになる

マージキューはこれを自動化する。

### 効果

- マージ承認後の待ち時間がほぼ消える (キューに入れて手放せる)
- main が常に「CI green な状態」に保たれる
- 1 日のマージ件数が増える (大規模チームほど効く)

## 仕組み

### 基本動作

```
1. PR が承認される → "merge when ready" に入れる
2. キューに並ぶ
3. キューの先頭の PR を最新 main にリベース (自動)
4. CI 実行 (自動)
5. CI green なら main にマージ
6. キューの次の PR に進む (3 に戻る)
```

CI 失敗時は PR を著者に返す (キューから外す)。

### 並列処理

大規模チームでは「シーケンシャルに 1 個ずつ」だとキューが詰まるので、複数 PR を投機的に並列実行できる:

- A, B, C を「もし A B C 全部成功したら main = X + A + B + C」と仮定して並列 CI
- 全部 green ならまとめてマージ
- A だけ失敗したら、B C を「main + B + C」で再実行

これにより 1 PR の CI 時間がボトルネックになっても、スループットを上げられる。

## 具体的なツール

| ツール                 | 特徴                                                                  |
| ---------------------- | --------------------------------------------------------------------- |
| **GitHub Merge Queue** | GitHub 純正 (2023 GA)。Enterprise / Team プランで利用可能。設定が簡単 |
| **Aviator**            | 商用。並列実行・優先度・カスタムルールが強い                          |
| **Trunk.io**           | 商用。CI 時間最適化と組み合わせ                                       |
| **Mergify**            | 商用 + OSS。ルールベースで柔軟                                        |
| **Bors-NG**            | OSS。Rust の bors の系譜。シンプル                                    |

**推奨**: GitHub Merge Queue から始める。プラン上使えるなら追加コストなしで導入できる。スループット要件が上がってから商用ツールを検討する。

## セットアップ要件

- **CI が並列実行可能** であること (シリアル実行だとキューが詰まる)
- **CI が安定していること** (flaky テストがあるとキューが頻繁に止まる)
- **保護ルールでマージキュー必須化** を設定する (キュー経由しか main にマージできなくする)
- **マージ戦略の決定**: squash / rebase / merge commit のどれにするか

## 落とし穴

| アンチパターン           | 何が問題か                                                              |
| ------------------------ | ----------------------------------------------------------------------- |
| flaky テストを放置       | キューが頻繁に止まる。安定化が前提                                      |
| CI が遅すぎる            | キューがボトルネックになって従来より遅くなる。CI 時間 < 10 分が望ましい |
| 全 PR をキュー必須にする | 緊急 hotfix が出せなくなる。例外パスを設計する                          |
| 並列度を上げすぎる       | 投機実行が外れたときの再実行コストが大きい。3-5 並列が無難              |
| 段階的 PR との順序衝突   | stacked PR は層単位で順序を守る必要がある。設定で明示する               |

## upflow での扱い

upflow はマージキューを利用していない (2026年5月現在)。

- 理由: チーム規模が小さく、マージレースが起きていない
- 利用したくなる兆候: 1 日 10 PR 以上のマージで、レビュー → リベース → CI 再実行が頻発したとき
- 計測: マージキュー経由かどうかを upflow で計測する機能はない (現状は対象組織が利用していない)

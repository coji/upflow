# テスト自動化

## 要点

DORA の定義: 「自動テストと手動テストを開発ライフサイクル全体で継続的に実行し、機能・アーキテクチャを検証して欠陥を早期に捕まえる」。手動テストはリリース頻度の根本的なボトルネック。AI 時代に大量のテストが書けるようになったが、保守コストとシグナル品質が新たな論点。

## なぜ重要か

### 手動テストはリリース頻度を殺す

DORA の説明そのまま: 「自動テストは数分で結果を返せる。手動テストは数日〜数週間かかる」。リリース頻度を上げたければ、テストが手動である限り physical limit に当たる。

### テストが土台になる理由

`delivery/` の他の能力 (trunk-based / CI / CD / deployment automation) は **「テストが速く、安定し、信頼できる」** ことを暗黙の前提にしている。テストが遅い・不安定だと:

- CI が 30 分かかる → daily merge が崩れる ([continuous-integration.md](./continuous-integration.md))
- flaky テストで CI が止まる → マージキューが詰まる ([../pr-flow/merge-queue.md](../pr-flow/merge-queue.md))
- テストの信頼性が低い → デプロイ前手動確認が増える → Deploy Time が伸びる

つまりテスト自動化は他のすべての前提条件。

## DORA の推奨実践

### テストピラミッド

下から:

1. **Unit tests** (基底): 個別関数・メソッドを単体で検証。**最も多く書く**。実行速度が最速
2. **Acceptance tests** (中間): 機能単位の検証。アプリケーションを動かして API / UI を叩く
3. **Manual testing** (頂点): 探索的テスト・ユーザビリティ確認。**最少**

「速いテストで先に欠陥を捕まえる」のが原理。

### TDD (Test-Driven Development)

開発者が **本番コードより先にユニットテストを書く**。DORA は推奨実践として明示している。利点:

- 設計の指針になる (テストしにくいコード = 設計が悪い)
- カバレッジが自然に高くなる
- リファクタの安全網になる

### 速度の絶対基準

- ローカルでも CI でも **10 分以内** で feedback が返ること
- これを超えると開発者がテストを skip し始める

## DORA のアンチパターン

| 失敗                                             | なぜ問題か                                       |
| ------------------------------------------------ | ------------------------------------------------ |
| テストを開発者の責任から切り離す (QA チーム任せ) | 開発者がテスト前提でコードを書かなくなる         |
| テストスイートのキュレーションを怠る             | 古いテスト・冗長なテストが溜まり実行時間が伸びる |
| flaky なテストを許容する                         | 信頼性が下がり、結局誰も結果を見なくなる         |
| mock に過度依存                                  | 実際の統合点で壊れるが気づけない                 |
| テスト比率を間違う (e2e 偏重)                    | 速度と安定性の両方を失う                         |

## AI 時代の論点

AI で**テストを大量に書ける**ようになった。これは両刃。

### 利点

- カバレッジを短時間で上げられる
- 境界値・異常系を網羅的に出せる
- レガシーコードへのテスト追加コストが下がる

### 新たな問題

| 問題                       | 内容                                                                                                |
| -------------------------- | --------------------------------------------------------------------------------------------------- |
| **保守コスト**             | AI 生成テストは大量に出るが、本番コードが変わるたびに更新が必要。生成して放置すると flaky の温床    |
| **シグナル品質の劣化**     | 「テストは通るが意味のあることをテストしていない」が増える。assert 文が浅い、何でも mock している等 |
| **テストピラミッドの破壊** | AI は e2e を生成しがち (UI シナリオが書きやすい)。気づいたら逆ピラミッドになっている                |
| **TDD の意味の希薄化**     | AI が後付けでテストを生成すると、テスト → 設計のフィードバックが消える                              |

### 推奨

- **AI 生成テストは PR レビュー対象**: 機械生成だからといってスキップしない。質を人間がチェック
- **生成元の commit に紐付ける**: 後で「なぜこのテストが存在するか」を追えるように
- **ピラミッド比率の監視**: unit / integration / e2e の数を週次で見る。逆転していたら警告
- **flaky 監視を導入**: 生成テストの flaky 率は手書きより高い傾向。retry 回数を計測する

## 計測

DORA の計測項目:

- 開発者が書いたテストの割合
- テスト段階別の欠陥検出数
- テスト失敗の修正時間
- テスト失敗が意味のあるシグナルか
- パイプラインへの統合状況

加えて AI 時代では:

- AI 生成テストの保守頻度・削除頻度
- flaky テスト率 (AI 生成と手書きで分けて)

## upflow での扱い

- ✅ Vitest (unit テスト): `app/` `batch/` 配下に `*.test.ts`
- ✅ Playwright (E2E): `pnpm test:e2e`
- ✅ `pnpm validate` で全テスト + lint + typecheck + build
- ✅ TDD は強制してないが、機能追加時に test 必須 (`CLAUDE.md` の PR チェックリスト 4)
- ⚠️ テスト実行時間: ローカル 5 秒程度 (validate 全体で約 30 秒)。10 分基準は満たす
- ❌ flaky 監視: 仕組みなし
- ❌ AI 生成テストの方針: ガイドラインなし
- ❌ ピラミッド比率の継続監視: 仕組みなし

特に **AI 生成テストの方針** は今後 AI で生成する場面が増えるので明文化の必要あり。

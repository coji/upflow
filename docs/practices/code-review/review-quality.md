# レビューの質を保つ仕掛け

## 要点

レビューが「速い」だけでは意味がない。指摘の質、著者との対話、認知負荷の管理を仕組みで支える。AI が下書き品質のコードを大量に出す時代、人間レビューの重みは「設計判断と意図確認」に偏っていく。

## なぜ重要か

### 速いだけのレビューの罠

- LGTM だけ返すレビュー → 著者は学ばない、欠陥が残る
- 大量の nit 指摘 → 重要な指摘が埋もれる、修正ループが増える
- 著者を萎縮させるトーン → 心理的安全性が下がる、PR 投稿頻度が下がる

### AI 時代の文脈

- AI が書いたコードは「動くが文脈に合わない」が起きやすい
- 人間レビュアーは AI が見ない「組織固有の文脈」を担保する役割
- 認知負荷が増えるので、レビュー対象を絞ることが重要 ([ai-human-split.md](./ai-human-split.md))

## 具体的な実践

### 1. PR description が起点

著者は PR を出す前に description に以下を書く:

- **なぜ** この変更が必要か (背景、リンクする issue / RDD)
- **何を** 変えたか (要点を箇条書き)
- **どう** テストしたか (手元での確認、追加したテスト)
- **影響範囲** (リスクの高い変更ならここで注意喚起)

description が薄いとレビュアーが context を集める時間が伸びる。著者の 5 分でレビュアーの 30 分を節約する。

### 2. レビュー粒度の階層

指摘には種類があり、優先度を区別する:

| 種類         | 例                                 | 扱い                             |
| ------------ | ---------------------------------- | -------------------------------- |
| **Blocker**  | バグ、セキュリティ、設計の根本問題 | マージ前必須対応                 |
| **Major**    | 保守性、テスト不足、ドキュメント   | マージ前推奨、合意あれば後回し可 |
| **Minor**    | 命名、リファクタ余地               | マージ後 follow-up でも可        |
| **Nit**      | スペース、コメント微修正           | 著者裁量                         |
| **Question** | 理解確認                           | 質問だけ、対応不要               |

レビュアーは prefix (`[Blocker]`, `[Nit]` 等) を付けて、著者が優先順位を判断できるようにする。

### 3. 質問と指摘を分ける

「これって XXX のためですか?」と「これは YYY すべきです」は違う。質問は学習機会、指摘は判断要請。混ぜると著者が混乱する。

### 4. レビューチェックリスト

組織で標準化したチェックリストを持つ:

- [ ] PR サイズが 400 行以内 (例外は理由を明示)
- [ ] テストが追加または更新されている
- [ ] description に「なぜ」が書かれている
- [ ] 関連する RDD / issue がリンクされている
- [ ] AI レビュー指摘に対応または却下している
- [ ] 既存の規約から逸脱していないか
- [ ] エラーハンドリングが境界 (外部入力) で適切か

### 5. レビューコメントのトーン

- 主語を「コード」にする (「あなたが」ではなく「このコードが」)
- 提案には根拠を添える (「こうするとパフォーマンスが…」「保守時にこう困る…」)
- 質問形を活用 (「これって…?」)
- 同意できる点もコメントする (LGTM 部分の明示)

### 6. 修正ループの最小化

レビュアー側:

- まとめて指摘を返す (1 つずつ pull で push が増えるのを避ける)
- 主要指摘の対応案も併記する (著者が手戻り少なく直せる)

著者側:

- 指摘への対応をコミットメッセージで明示
- 異論があれば反論する (即受け入れない)

## 落とし穴

| アンチパターン                                | 何が問題か                                                                                                    |
| --------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| Nit ばかり指摘                                | 重要指摘が埋もれる。Blocker / Major と区別する                                                                |
| 黙って force-push で対応                      | レビュアーが追跡困難。reply + commit のセットで                                                               |
| 「これでいい?」と結論を著者に丸投げ           | レビュアーの判断が必要なときは明確に                                                                          |
| LGTM だけで承認                               | 学習機会の喪失。最低限「ここが良い」「ここが気になった」を書く                                                |
| 大規模 PR を「全部読む」と諦めて流す          | 結果的にレビューなしマージと同じ。上限規律 ([pr-size-discipline.md](../pr-flow/pr-size-discipline.md)) で防ぐ |
| AI レビュー指摘を全部レビューコメントにコピー | ノイズ。AI と人間の領域を混ぜない                                                                             |

## 認知負荷の軽減

レビュアーの認知負荷を下げる仕組み:

- **PR を小さく保つ** ([pr-size-discipline.md](../pr-flow/pr-size-discipline.md))
- **段階的 PR で文脈を分ける** ([stacked-prs.md](../pr-flow/stacked-prs.md))
- **AI レビューで lint / 機械チェックを終わらせる** ([ai-human-split.md](./ai-human-split.md))
- **PR description で前提を共有**
- **コミットを論理単位で分割** (1 commit = 1 思考単位)

## 計測

質の高いレビューが回っているかの指標:

| 指標                           | 望ましい傾向                                |
| ------------------------------ | ------------------------------------------- |
| 1 PR あたりレビューコメント数  | 多すぎず少なすぎず (3-10 が目安)            |
| レビュー往復回数               | 1-3 回が理想。5 回超は要因分析              |
| 修正後のレビュー応答時間       | 初回より速いことが多い (短い差分なので)     |
| レビュー指摘起因の不具合検出率 | 高いほど質が良い (本番障害の事後分析で測る) |

数字を主目的化しない。「コメント 0 で承認なし」と「コメント 30 で改善多数」は前者がコメント数だけ少ない。

## upflow での扱い

upflow はレビュー件数・レビュアーは raw データで持つが、レビューの「質」は計測していない。

- ✅ レビュー件数 (`pull_request_reviews`)
- ✅ レビュアー (`pull_request_reviewers`)
- ❌ レビューコメント数 (raw データに無し、GitHub API 取得で追加可能)
- ❌ レビュー往復回数 (commit 後の re-review を識別する集計が必要)
- ❌ レビュー指摘起因の障害検出率 (本番障害との接続が必要)

質の計測は将来課題。まずは速度と量の計測を充実させる方針。

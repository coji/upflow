# AI と人間の役割分担

## 要点

AI レビューは「機械的にチェックできるもの」を担当し、人間は「設計判断と意図」に集中する。両方を併用したチームでレビュー時間 62% 削減の事例が報告されている。AI レビューだけ・人間レビューだけのどちらに振り切っても効かない。

## 役割分担表

| 領域                     | AI が得意                    | 人間が得意             |
| ------------------------ | ---------------------------- | ---------------------- |
| **構文・スタイル**       | ◎ lint, format               | △ 不要                 |
| **セキュリティパターン** | ◎ 既知脆弱性パターン検出     | ○ 新規脅威モデル       |
| **テスト網羅性**         | ◎ カバレッジ・境界値         | ○ ドメイン特有のケース |
| **クロスリポジトリ影響** | ◎ 大規模 grep                | △ 限定的               |
| **リスク分類**           | ◎ 変更の危険度ラベリング     | ○ 文脈含む判断         |
| **アーキテクチャ判断**   | △ ガイドラインの照合は可     | ◎ トレードオフ判断     |
| **意図と実装のズレ**     | △ コメント vs コード対比は可 | ◎ 仕様理解             |
| **長期保守性**           | △ 表面的な複雑度のみ         | ◎ 将来予測             |
| **ドメイン知識**         | ×                            | ◎                      |
| **エッジケース推論**     | △ 既知パターンのみ           | ◎                      |

## なぜ重要か

### AI 単独では足りない

- **意図** はコードに書かれていない (コミットメッセージや仕様書を踏まえても完全には埋まらない)
- **アーキテクチャ判断** は組織固有の文脈に依存する (なぜこの抽象を選んだか)
- **長期保守性** は AI の評価基準が曖昧

### 人間単独では追いつかない

- AI 時代の PR 数増加 (DORA 2025: +98%) で、人間レビュアーの容量が枯渇
- lint / セキュリティ / カバレッジは人間がやるべき仕事ではない
- 大規模リポジトリでのクロス影響は人間には見えない範囲がある

### 併用の効果

Cloudflare の事例では、AI レビュー (CodeRabbit) を導入後、人間レビュー時間 **62% 削減**、修正ループ回数の減少、初回 PR 品質の向上が報告された。

## 具体的な実践

### 1. AI レビューの位置付け

- PR を開いた直後に AI レビューが走る
- AI 指摘を著者がまず修正してから人間レビュー依頼
- これにより人間レビュアーは「lint 直して」を言わなくて済む

### 2. AI レビューツール

| ツール                         | 特徴                                |
| ------------------------------ | ----------------------------------- |
| **CodeRabbit**                 | 包括的。日本語対応。upflow で利用中 |
| **Greptile**                   | 大規模リポジトリの context が強い   |
| **Qodo**                       | OSS 由来。ルールカスタマイズ柔軟    |
| **GitHub Copilot Code Review** | GitHub 純正。Copilot 契約に含まれる |
| **Codex Code Review (OpenAI)** | OpenAI 純正。新しめ                 |
| **Cursor Bugbot**              | Cursor 統合。エディタ内で完結       |

ツール選定は (1) 既存の CI / レビューワークフローへの統合のしやすさ、(2) 言語サポート、(3) コスト、で判断。

### 3. 人間レビューの集中領域

レビュアーは AI 指摘を見ずに、以下に集中する:

- **設計判断**: なぜこの抽象を選んだか、別の案はあったか
- **意図のズレ**: PR description と diff が一致しているか
- **隠れた前提**: 「この前提が崩れたら壊れる」を指摘
- **テストの妥当性**: テストが要件を反映しているか (網羅率は AI で十分)

### 4. 役割分担の文書化

- レビューチェックリストに「AI が見るもの / 人間が見るもの」を明記
- 新メンバーへのオンボーディング資料に含める
- レビューコメントで「これは AI が指摘済み」と書かない (ノイズになる)

## 落とし穴

| アンチパターン                         | 何が問題か                                                          |
| -------------------------------------- | ------------------------------------------------------------------- |
| AI 指摘を全部スルー                    | せっかくの AI レビューが死蔵される。最低限「対応 / 却下」を明示     |
| AI 指摘を全部受け入れる                | AI も誤検出する。判断は著者が持つ                                   |
| 人間が AI 指摘を再確認する             | 二度手間。AI の領域は AI に任せる前提で                             |
| AI レビューを承認カウントする          | 政策上 AI は人間と同じ責任を持てない。承認は人間のみ                |
| AI レビューに細かい指摘を求めすぎ      | "10 個の nit を全部書いて" 系は逆にノイズ。優先順位を AI 設定で絞る |
| AI レビューの差分を毎回 0 から走らせる | 大規模 PR で時間がかかる。差分のみのモードを使う                    |

## 計測

役割分担が機能しているかを見る指標:

| 指標                         | 望ましい傾向                                       |
| ---------------------------- | -------------------------------------------------- |
| AI レビュー指摘の対応率      | 50% 以上 (低すぎたら AI 設定が合っていない)        |
| 人間レビュー時間 (PR あたり) | 導入前比で減少                                     |
| 人間レビュー指摘の質         | 「設計」「意図」関連が増え、「lint」「typo」が減る |
| 修正ループ回数               | 導入後に減少                                       |

## upflow での扱い

upflow は CodeRabbit を AI レビューに利用しており、Codex / Cursor も補助的に使う運用 ([CLAUDE.md](../../../CLAUDE.md) の PR 前 self-review 参照)。

- ✅ レビュー件数・著者は raw データで保持
- ❌ AI レビュー (bot 投稿) と人間レビューの分離計測は未実装
- ❌ AI 指摘の対応率の計測は未実装 (CodeRabbit の resolve 機能と連携が要る)
- ❌ レビュー時間の AI / 人間別集計は未実装

issue #332 の RDD で「bot レビューと人間レビューの分離」を要件に入れる余地がある。

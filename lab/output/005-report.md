# LLM分類ベースのオートマージ効果レポート

> 実験005 — 2026-02-23 / 対象: 2025年以降のマージ済みPR 971件

## Executive Summary

ルールベース（行数・ファイル数）のPR分類をLLM（Gemini 2.5 Flash Lite）に置き換えることで、**XS+S自動マージによるキュー削減効果が19%→50%に劇的に改善**する。コストは全PR分類で約$0.06（PR単価$0.00017）と実用的。

---

## 1. 分類精度の問題（実験004の知見）

ルールベース分類器は`additions + deletions`と`changedFiles`のみで判定するため、以下の**構造的な偽陽性**が発生する：

| パターン                           | 例                                            | ルール | LLM | 正解 |
| ---------------------------------- | --------------------------------------------- | ------ | --- | ---- |
| テストデータ変更が大量             | README・テストデータ整理 (+214/-242, 19files) | L      | S   | S    |
| 削除中心のリファクタ               | セッション種別検出変更 (+77/-339, 9files)     | L      | S   | S    |
| ドキュメント・設定ファイル大量追加 | BMAD設定 (+65,443, 340files)                  | L      | M   | M    |
| ファイル数は多いが表記ゆれ修正     | 表記ゆれ統一 (+36/-28, 11files)               | L      | XS  | XS   |

サンプル20件（ルールL判定のみ）で**一致率40%**。不一致10件を全件ファイル内容で検証した結果、**全てLLMの方が妥当**だった。

## 2. サイズ分布の変化

| サイズ | ルールベース | LLM         | 差分     |
| ------ | ------------ | ----------- | -------- |
| **XS** | 300 (30.9%)  | 471 (48.5%) | **+171** |
| **S**  | 131 (13.5%)  | 211 (21.7%) | **+80**  |
| M      | 379 (39.0%)  | 202 (20.8%) | -177     |
| L      | 160 (16.5%)  | 70 (7.2%)   | -90      |
| XL     | 1 (0.1%)     | 17 (1.8%)   | +16      |

LLM分類では**全PRの70%がXS/S**に分類される（ルールベースでは44%）。
ルールベースがM/Lに過剰分類していたPRの多くが、LLMによりXS/Sに正しく再分類された。

## 3. オートマージシミュレーション結果

| シナリオ      | ルールベース avg | LLM avg | ルール削減率 | LLM削減率  |
| ------------- | ---------------- | ------- | ------------ | ---------- |
| Baseline      | 13.1             | 13.1    | —            | —          |
| Auto XS       | 11.4             | 11.5    | -13.4%       | -12.4%     |
| **Auto XS+S** | **10.6**         | **6.6** | **-19.1%**   | **-50.0%** |

**XS+S自動マージの効果がルールベースの2.6倍**に向上。
LLM分類によりSに正しく分類されるPRが80件増え、自動マージ対象が大幅に拡大したため。

## 4. LLMサイズ別レビュー待ち時間

| サイズ | n   | median | p75    | p90    |
| ------ | --- | ------ | ------ | ------ |
| XS     | 219 | 0.5h   | 11.0h  | 46.4h  |
| S      | 157 | 4.0h   | 28.7h  | 113.2h |
| M      | 136 | 6.6h   | 69.2h  | 162.4h |
| L      | 42  | 23.3h  | 115.3h | 167.4h |
| XL     | 4   | 79.9h  | 137.0h | 137.0h |

LLM分類のサイズと実際のレビュー待ち時間がきれいに相関しており、**分類の妥当性を裏付け**ている。
XSのmedian 0.5hは「すぐ承認されるPR」、XLのmedian 79.9hは「実際にレビューが重いPR」に対応。

## 5. コスト分析

| 項目                     | 値                    |
| ------------------------ | --------------------- |
| モデル                   | gemini-2.5-flash-lite |
| 全971件のコスト          | **$0.06**             |
| PR単価                   | $0.00017              |
| 月次運用想定（100PR/月） | $0.02/月              |

Gemini Flash Liteの価格（input $0.075/1M, output $0.3/1M tokens）により、実運用でもコストは無視できるレベル。

## 6. アクショナブルな提案

### 短期（Upflow機能として実装可能）

1. **LLMベースPRサイズ分類の導入**
   - ルールベースを置き換え、PR作成時にGemini Flash LiteでXS〜XLを判定
   - コストは無視できるレベル（$0.02/月）
   - `risk_areas`と`suggested_reviewers`も副産物として取得可能

2. **XS/S PRの自動マージ推奨バッジ**
   - LLM分類でXS/Sと判定されたPRに「Auto-mergeable」バッジを表示
   - レビュアーに「このPRは軽い」というシグナルを送る
   - **期待効果: レビューキュー50%削減**

### 中期（チーム運用改善）

3. **サイズ別SLAの設定**
   - XS: 当日中、S: 翌営業日、M: 2営業日、L: 3営業日
   - 現状のp75を基準にすると実現可能なライン

4. **XL PRの分割推奨**
   - LLM分類でXLと判定されたPRに分割を推奨
   - 現状17件（1.8%）のみだが、median 79.9h待ちは改善余地大
